# 目标偏移分析（2026-07-01 回看）

> 回看时机：L0→L4 全流程骨架落地 + 6 个 OpenSpec change archived + `l4b-docker-run` 端到端实跑验证笔记完成之后。
> 回看视角：对照两个核心需求（需求 A 发现 / 需求 B 研判），审视开发是否偏移目标。
> 本文是一份**基于文件系统实证**的偏移记录，不是设计方案。设计稿仍是 `total-design.md`（第一参考源）。

## 0. 两个核心需求（出处：total-design §1.1）

- **需求 A（快）**：在几千只 A 股里快速找到低估值/低 PE/低热点的票布局入场。
- **需求 B（深）**：多 agent 分析师天团，通过辩论达成共识，判断标的在当前节点好不好、为什么、什么条件下会改变——这是仓位决策的前置判断，用户结合自身持仓状态做最终的加仓/持有/减仓/清仓决策。

翻译成两个核心问题：
1. **发现**——从 ~5000 只 A 股中找到值得关注的标的（L1+L2 快筛管线）
2. **研判**——对已关注的标的做深度质性判断，辅助买卖决策（L3 天团辩论管线）

第一性原理修正（§1.2）：「低热点」是排除维度不是反转因子；「合适时点入场」重定义为「等待基本面催化」，不是 K 线择时。

- 6 份空壳只有 295 字节，`key_variables` 仅 1 条且全是"市场份额大幅下降"这句复读。
- 唯一真实产出是 600009（上海机场，门 2 跑的）：1974 字节，8 条 key_variables、有 dissent（冯柳逆向）、5 条 pending_verification。

### 1.2 L3 同质化 + 串台：比预想更严重

读 `debate/600519/2026-07-01.md`（茅台，门 1）原文：

- **串台（环形引用，非随机泄露）**：R1 本该隔离（`other_opinions=None`），但 4 个 agent 的 `core_thesis` 互写别人，且构成**有方向的环**：

  ```
  buffett  → "munger 看好长期价值"
  munger   → "duan 看好长期价值"
  duan     → "feng_liu 看好长期价值"
  feng_liu → "buffett 重新审视后仍看好"
  ```

  环的方向（buffett→munger→duan→feng_liu→buffett）与 agent 注册顺序一致——这不是随机串台，暗示根因在 prompt 模板（含示例/占位文本被复读），而非 `other_opinions` 运行时泄露。
- **逐字同质化**：4 个 agent 的 `key_metrics`（ROE 32%、毛利率 90%+）、`risks`（估值偏高）、`historical_parallel`（可口可乐）、`what_would_change_my_mind`（市场份额大幅下降）**完全相同**。
- **无哲学差异**：4 个 conviction 都是 75，R2 微调到 77/79/81/0，没有投资哲学分野。

### 1.3 单 agent 跳过 + 复读污染（同一份输出，非同质化）

读 `debate/600900/2026-07-01.md`（长江电力）原文：

- R2/R3/R4 明文写「（单 agent 模式，跳过）」——600900 只跑了 buffett 一个 agent。
- **致命证据**：600900 单 agent 模式下，buffett 的 R1 输出与 600519 的 buffett R1 输出**逐字相同**，包括 `conviction=75`、`ROE 32%`、`毛利率 90%+`、`可口可乐`、`市场份额大幅下降`。600900 是水电股，ROE 根本不是 32%、毛利率也不是 90%+。**这不可能是两次独立 LLM 调用的结果**——是同一份输出被复读。

**三个并行排查的根因假设**（按可能性排序，主线在最前）：

1. **【主线·最可能】prompt 模板含示例/占位文本被复读**——R1 模板里可能有一段示例串（形如 `"{other_agent} 看好长期价值"`），当 features 没正确注入时，模型直接套用模板示例。这个假设能**同时**解释三个现象：
   - 环形引用（§1.2）：示例串本身是环，每个 agent 套一句
   - 600900 复读茅台：features 没进去，只剩模板示例，两只票输出相同
   - 同质化：示例是一套，所有 agent 都复读它
   - 单 agent 模式下 buffett 仍写"munger 看好"：600900 根本没跑 munger，这句话只能来自 prompt 本身，不可能来自 `other_opinions` 运行时泄露
2. **features 注入空/错**——`_build_user_message()` 传入的 `features` 对 600900 返回空，或因 ticker 后缀不一致读到 600519 的缓存数据。但纯数据缺失解释不了"munger 看好"这种**引用别人的元叙述**（数据里没有这种话），故为次要假设。
3. **LLM 响应被缓存**——`llm.py` 若有响应缓存层，相同 prompt→相同响应。但缓存假设解释不了"4 个 agent 各生成一句**不同**的环形引用"（缓存复读应是 4 句完全相同），故可能性最低，仍需排查以排除。

**排查优先级**：先查 `council/prompt.py` 的 R1 模板构造（假设 1），再查 `council/debate.py` R1 调用 + `_build_user_message()` 的 features 注入（假设 2），最后查 `council/llm.py` 是否有响应缓存层（假设 3）。这是可定位的 bug，不是「模型能力不足」的玄学。

### 1.4 cache ticker 后缀分裂

`data/cache/` 下 26 个 ticker 目录，同票两份并存：

| 纯数字目录（有真实数据） | 带后缀目录（多为空壳） |
|---|---|
| `600519/`（basic+financials+kline+risk 齐全） | `600519.SH/`（只有空的日期子目录 6/24-6/27） |
| `600900/`（5 维全） | `600900.SH/`（空） |
| `000858/`（5 维全） | `000858.SZ/`（只有空日期子目录） |

下游影响：`screen` 用纯数字读得到数据，`council` 用 `.SH` 读空壳 → 门 1 首跑 `insufficient_data` 崩溃的根因之一（叠加 TTL 过期）。

### 1.5 全市场从未跑过

- `data/cache/` 26 个 ticker 目录全是手工挑的样本（茅台/平安/五粮液/长江电力等白马），**不是从 5000 只里筛出来的产物**。
- `review-notes.md` 门 2 跑的 `batch data/tickers.txt` 只有 20 只手工清单。
- L1 的 hard gates / heat filter **从没见过全市场分布**，阈值合理性、5000→200 漏斗比例、`industry_pe_degraded` 在全市场触发面——全未知。
- `stats.input_scale == "subset"`（20 < 300）这个退化标记本身就在说「这不是真·全市场」。

## 2. 偏移判断

### 2.1 大方向没偏

骨架与设计稿高度对齐，设计稿明确砍掉的领域都没有跑偏进去：
- 两条管线解耦（AD-01）✅
- L2 成本闸门（AD-03）✅ 代码层面有
- 格雷厄姆在 L1 内核（AD-07）✅
- 不做回测 / 不做择时 / 砍技术派（AD-02/AD-06）✅
- 不用 Multi-Agent 框架（AD-05）✅

### 2.2 两个核心需求都只落实了「代码骨架」，没落实「能力」

| 维度 | 代码骨架 | 端到端验证 | 能交付价值吗 |
|---|---|---|---|
| **需求 A（发现）** | ✅ 完整 | ❌ 零（从未全市场跑过） | ❌ 不能 |
| **需求 B（研判）** | ✅ 完整 | ⚠️ 1/7 真跑通，6/7 空壳+同质化 | ❌ 不能（质量极不稳定） |

### 2.3 两个核心架构假设的佐证状态

| 假设 | 出处 | 佐证状态 |
|---|---|---|
| L2 是必要的成本闸门（200→20，¥0.01/只） | AD-03 / §3.1 | ❌ 零佐证。从未在真实候选集上验证筛选率和区分度 |
| L3 4 轮辩论产生信息增量 | AD-05/AD-09 | ⚠️ 1 个佐证（600009：多数中性 vs 冯柳逆向），6 个反例（同质化+串台）。**好结果是偶然，坏结果是常态** |

### 2.4 开发顺序偏移

设计预期（§八）：L1+L2 先行实跑验证 → L3 从单 agent 起步验证辩论增量（AD-09 gate）→ 全天团 → L4 最后。

实际执行：
- L0-L2 代码写完，**没实跑**
- L3 单 agent gate 写了（`len(agents)==1` 跳过逻辑在），**但没真用 gate 卡住就往前走**
- 全天团代码写完，**6/7 产出同质化**
- L4 + Docker 已归档 ✅

**7 个 archived change 全部是「构建」类，零个「验证」类 change。** 基础设施（Docker、L4 周报/告警）花了大量精力，但核心链路没跑通。L4 监控在 watchlist 6/7 全 null 的情况下无意义。

### 2.5 需求 B 语义已澄清（设计层，非偏移，已关闭）

设计稿 §1.1 原文：「给**已有仓位**做下一步决策」。2026-07-01 澄清：

- L3 的职责是判断「这只票在当前节点好不好、为什么、什么条件下会改变」——这是仓位决策的**前置判断**
- 用户拿着 L3 的判断 + 自己的仓位信息（成本/仓位权重/组合风险）做最终的加仓/持有/减仓/清仓决策
- 当前 L3 schema（`final_verdict` + `conviction` + `consensus_summary` + `dissent_points` + `key_variables` + `pending_verification`）**语义正确，不需要改**

理由：
- 仓位信息是用户知道的，不是系统知道的——成本价、仓位占比、整体组合风险敞口，维护成本高，MVP 阶段硬塞进来反而是负担
- 「好不好」是客观判断，「怎么办」是主观决策——投资哲学分歧（巴菲特 vs 冯柳）应该在「好不好」上体现，不应被仓位噪音污染
- 600009（上海机场）那份产出已证明这个模式能用：neutral + 8 条 key_variables + 冯柳 dissent，拿着这个 + 自己的仓位信息完全可以做下一步决策

**P2 已关闭。**

## 4. 纠偏优先级

### P0 — L3 R1 串台+同质化根因定位（最可疑，可直接动手）

**为什么优先**：需求 A 的「没跑过」是工作量问题（L1 纯 Python 零 LLM，跑出来大概率能筛出东西，风险低）；L3 同质化是**假设存亡问题**——如果 4 个 agent 就是会输出一样的东西，「天团辩论」立项前提就是错的，整个需求 B 要重新设计。这是更上游的风险。

**最可疑线索**：600519（茅台）和 600900（长江电力）两只完全不同的票，R1 输出**逐字相同**（含 conviction=75）。600900 是水电股，ROE 根本不是 32%、毛利率也不是 90%+，却输出了「ROE 32%、毛利率 90%+、可口可乐、市场份额大幅下降」。这不可能是两次独立 LLM 调用的结果。

**定位方向**（bug 定位，非改设计，三假设按可能性排序，详见 §1.3）：
1. **【主线】查 `council/prompt.py` 的 R1 模板构造**——是否含示例/占位文本（如 `"{other_agent} 看好长期价值"`），features 缺失时被模型复读。此假设能同时解释环形串台、600900 复读、同质化三个现象。
2. 查 `council/debate.py` R1 调用 + `_build_user_message()` 的 features 注入——`other_opinions=None` 为什么还会串台、600900 的 features 是否读到空或读到 600519 缓存。
3. 查 `council/llm.py` 是否有响应缓存层——排除相同 prompt→相同响应的可能。

查清后能直接判定 AD-09 假设是否成立。

### P1 — 需求 A 全市场验证

- 修 cache ticker 后缀分裂（normalize 在 CacheManager 层做，统一纯 6 位数字 key）
- 用 akshare 拉全 A 股代码，跑一次 `batch` + `screen`
- 看 L1 在 5000 只分布下的漏斗比例（5000→?→?→?）和 `industry_pe_degraded`/`input_scale` 触发情况
- 跑 `scout` 看 L2 在真实候选集上的区分度（验证 AD-03 成本闸门假设）
- 可能暴露 L1 阈值需要校准的问题（§4.8 规划了权重校准但没做）

- P0 跑通后可直接进入大规模验证 AD-09，无需先改 L3 输入输出 schema。

## 5. 一句话总结

**骨架全搭完，但核心链路基本没真正跑过。开发重心过早滑向基础设施（Docker/L4 监控），两个核心架构假设（L2 成本闸门有效 + L3 辩论产生信息增量）零到极弱佐证。L3 同质化有一只反例铁证（600900 复读茅台特征），建议优先定位这个 bug 以判定 AD-09 假设是否成立。**
