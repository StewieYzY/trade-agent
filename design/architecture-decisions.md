## AD-01 · 两条独立管线：快筛与深研解耦

**状态**：已确认  
**来源**：total-design §1.1

### 问题

用户直觉认为「快选股」和「天团深研」是同一条管线的两个环节——选股之后交给天团研判，但天团太重（¥20-60/只 × 5000 只 = ¥100,000-300,000），导致选股不可能快。

### 决策

快和深是**两条独立管线**，不是同一条管线的两个环节：

- **快筛管线**（L1 + L2）：全市场 ~5000 只 → 候选池 ~200 → 深研池 ~20。纯排序问题，规则能答。
- **深研管线**（L3）：对已有仓位或高优先级候选做天团辩论。质性判断问题，agent 才能答。

两条管线通过 **watchlist 接口**连接，不直接耦合。

### 对各 change 的约束

| Change | 约束 |
|--------|------|
| L1, L2 | 必须能独立运行并产出 watchlist，不依赖 L3 |
| L3 | 输入是单只股票的特征数据，不假设来自 L1/L2（可从手动输入） |
| L4 | 同时消费 L1/L2 的 watchlist 和 L3 的深研结果 |

## AD-02 · 「不择时」的第一性原理修正

**状态**：已确认  
**来源**：total-design §1.2

### 问题

用户原始需求包含「找低估值、低 PE、**低热点**的股票想**布局入场**」。前两个（低估值/低 PE）是价值投资正道，但后两个必须修正：

1. **「低热点」不创造价值**：无人问津的票可能是真低估（机会），也可能是价值陷阱（夕阳/造假/治理崩坏）。低热度本身不区分两者。
2. **「合适时点入场」是择时**：价值投资的立论基础是「不择时」——格雷厄姆说市场先生是来伺候你的不是来指导你的。小白试图「等低点入场」90% 变成「涨了追高、跌了割肉」。

### 决策

- **低热度**：作为**排除维度**（剔被炒的、剔刚炒完的），**不做反转打分因子**（跌多了 ≠ 机会）。
- **合适时点**：重新定义为「**等待基本面催化**」，不是 K 线低点。买入区间提醒 = 估值分位触及历史低位 + 出现基本面催化（财报超预期/行业政策/管理层换届/分红提升），不是「股价跌到某价位」。

### 对各 change 的约束

| Change | 约束 |
|--------|------|
| L1 | 低热度是排除维度（heat_filter），不是反转因子 |
| L4 | 估值区间提醒必须同时满足「估值低位 + 催化出现」两个条件，不允许只用估值低位 |

## AD-03 · 成本模型与 L2 成本闸门

**状态**：已确认  
**来源**：total-design §3.1

### 问题

L3 天团辩论成本高（¥20-60/只 × 多轮 × 5+1 agent），全市场 5000 只全跑 L3 成本不可承受（¥100,000-300,000/轮）。

### 决策

L1→L2→L3 是三层漏斗，每层成本递增、数量递减：

| 阶段 | 每只成本 | 数量 | 总成本 |
|------|----------|------|--------|
| L1 量化筛选 | ≈ 0 | 5000 → 200 | ≈ 0 |
| L2 LLM 初筛 | ≈ ¥0.01 | 200 → 20 | ≈ ¥2 |
| L3 天团深研 | ≈ ¥20-60 | 20 → 5 | ≈ ¥400-1200 |
| L4 监控 | ≈ ¥0.1 | 5-20 只/周 | ≈ ¥2/周 |

**L2 是必要的成本闸门**——没有它，200 只全丢 L3（¥4000-12000）不可承受。

### 对各 change 的约束

| Change | 约束 |
|--------|------|
| L2 | 必须实现，不允许跳过直接 L1→L3 |
| L2 | 成本目标 ¥0.01/只，使用轻量推理模型 |

## AD-04 · LLM 推理等级标注

**状态**：已确认  
**来源**：total-design §4.6

### 问题

系统有 5 个 LLM 调用场景，推理能力需求差异大，需要统一标注但不绑定具体模型。

### 决策

推理等级分三档，不固定模型种类，只标注该环节需要多强的推理能力：

| 推理等级 | 特征 | 适用环节 |
|----------|------|----------|
| **轻量** | 短输入短输出、分类/标注/结构化提取、不需要深度推理链 | L2 Scout（单股初筛） |
| **中度** | 中等长度输入、综合归纳/摘要、需要理解上下文但不需要独立推理 | L3 Round 4（收敛共识）、L4 触发重评估 |
| **重度 · 强思维链** | 长输入、跨维度关联推理、多立场模拟、对抗性分析 | L3 Round 1-3（各自表态/交叉质疑/DA 挑刺） |

**规律**：凡涉及「从数据做投资判断」或「跨 agent 辩论」→ 重度；凡做「分类筛选」或「综合归纳」→ 轻/中度。

### 对各 change 的约束

| Change | 约束 |
|--------|------|
| L2 | 使用轻量推理模型 |
| L3 | Round 1-3 使用重度推理模型，Round 4 使用中度 |
| L4 | 触发重评估使用中度推理模型 |

## AD-05 · 不用 Multi-Agent 框架（如 AgentScope/LangGraph）

**状态**：已确认  
**来源**：total-design §6.5

### 问题

L3 天团辩论是否应该使用 AgentScope/LangGraph 等 Multi-Agent 框架？

### 决策

**不需要，甚至不应该用。** 天团辩论的本质是「带上下文的串行 LLM 调用」，不是分布式多 agent 系统。

**关键差异**：

| 维度 | 多 agent 框架要解决的问题 | 我们的 L3 天团 |
|------|--------------------------|----------------|
| Agent 生命周期 | 长时间运行、自主决策、动态 spawn/kill | 单次调用，4 轮后结束 |
| 通信拓扑 | 动态路由、广播、点对点、订阅 | 固定拓扑：4 轮串行，每轮并行 5-6 个调用 |
| 工具调用 | 每个 agent 自主调用外部工具/API | 不需要——数据已由 L1/L2 准备好 |
| 状态管理 | 分布式状态、共享内存、一致性 | 无状态——每轮输入是上一轮的纯文本输出 |

**为什么用框架反而是坏事**：
1. 抽象泄漏：框架把简单的「调用 LLM → 拿到输出 → 拼进下一轮 prompt」包装成 Agent/Message/Tool/Channel 等概念，增加理解成本
2. 调试困难：agent 说了什么、为什么这么说、上下文在哪丢的——框架黑盒化后排查成本翻倍
3. 依赖风险：框架迭代快、API 不稳定，追版本消耗的精力远大于自己写 50 行编排代码
4. 过度约束：框架预设了 agent 通信模式，但我们的辩论编排可能需要频繁调整轮数、并行策略、信息过滤规则

**怎么做 A2A 通信**：`debate.py` 就是消息总线，不需要 agent 之间直接通信——辩论编排器负责收集、过滤、分发所有消息。

**什么时候才需要引入框架**（以下条件同时满足时再考虑）：
- agent 数量 ≥ 10 且动态增减
- agent 需要自主调用外部工具（不是只读数据）
- 通信拓扑不是固定的（agent 自主决定和谁对话）
- 需要持久化 agent 状态（跨会话记忆）

当前 MVP 不满足任何一条。

### 对各 change 的约束

| Change | 约束 |
|--------|------|
| L3 foundation | debate.py 是唯一的状态持有者和消息路由，不引入 AgentScope/LangGraph |
| L3 foundation | agent 间传递结构化 JSON（§6.3 AgentOutput schema），不传自由文本；结构化 JSON 比自由文本紧凑 5-10 倍，且 LLM 可直接消费无需再做摘要 |
| L3 foundation | 辩论记录每轮结束后立即写入 `debate/{ticker}/{date}.md`（append-only），而非 4 轮全部完成后一次性写入；确保中途崩溃或超时时已有部分记录可复盘 |
| L3 full council | 同上 |

## AD-06 · MVP 不做系统性回测

**状态**：已确认  
**来源**：total-design §10.3

### 问题

是否需要在 MVP 阶段做系统性回测（用历史数据验证 L1/L2/L3 的推荐质量）？

### 决策

**MVP 不做系统性回测。**

**理由**：
1. **案例库已覆盖基本校准需求**：用真实案例（茅台应该看多、乐视应该看空）校准 agent 判断质量（§6.6），这是 MVP 阶段的核心验证手段。
2. **L1 因子无需回测**：F-Score、PE/PB 分位、格雷厄姆数等都是学术验证过的公式（30 年+ 文献支撑），不需要自己回测有效性。
3. **系统性回测成本高、收益有限**：
   - L2/L3 的判断是质性推理，回测需要历史数据 + 历史标注（哪些股票后来涨了/跌了），数据获取成本高
   - 回测结果可能误导（历史不代表未来，过拟合风险）

**实际跑几轮后再评估**：看推荐质量、L2 否决率、L3 共识与市场走势的吻合度，决定是否需要系统性回测。

### 对各 change 的约束

| Change | 约束 |
|--------|------|
| L1 | 不要求回测验证，F-Score/PE/PB 等因子直接采用学术公式 |
| L2 | 一致性保障（§5.6）：temperature=0 + 阈值缓冲带 + 缓存 24h；不做案例校准 |
| L3 | 校准用真实案例（§6.6），不用历史数据集 |

## AD-07 · 格雷厄姆在 L1 规则引擎内核

**状态**：已确认  
**来源**：total-design §6.1（534-537 行）

### 问题

天团是 5+1 agent（巴菲特/芒格/段永平/冯柳/张坤 + Devil's Advocate），为什么没有格雷厄姆？

### 决策

**格雷厄姆不在天团里——他在 L1 规则引擎内核。**

格雷厄姆的贡献是**硬约束纪律**（PE×PB<22.5、7 项指标达标率、硬阈值），不是质性判断。把他的纪律嵌入 L1 的 Hard Gates 和格雷厄姆数，比把他做成一个 agent 更有效：

- agent 可能「觉得便宜但犹豫」
- L1 规则说「PE×PB>22.5 直接不进候选池」就不会犹豫

这是 L1/L3 职责边界的核心定义：
- **L1（格雷厄姆纪律）**：硬约束、硬阈值、一票否决、零主观判断
- **L3（天团质性判断）**：护城河评估、管理层判断、认知差识别、辩论收敛

### 对各 change 的约束

| Change | 约束 |
|--------|------|
| L1 | Hard Gates 必须内嵌格雷厄姆数（PE×PB<22.5）和格雷厄姆 7 项指标达标率 |
| L3 | 不做格雷厄姆 agent；格雷厄姆纪律已在 L1 强制执行，天团不重复判断 |

## AD-08 · 蒸馏从 Level 2 起步

**状态**：已确认  
**来源**：total-design §6.2（548-554 行）

### 问题

L3 agent prompt 应该怎么构造？有三个层级可选：
- Level 1（UZI 现状）：台词模板表 → 查表填空，不能辩论
- Level 2：规则 + 风格系统提示 → LLM 在风格内自由生成，可辩论
- Level 3（进阶）：知识库 RAG + 立场一致性约束

### 决策

**从 Level 2 起步**：一个 system prompt 内嵌该投资人决策框架 + 风格 + 喂入 stock_features 特征数据，让 LLM 在该投资人立场内生成论点、回应其他 agent。

- **不做 Level 1**：台词模板表不能辩论（UZI 的教训）
- **不做 Level 3**：RAG 是后期优化，MVP 先验证 Level 2 够不够用

Level 2 的 prompt 结构：
1. 核心决策框架（该投资人的判断逻辑）
2. 实际买过/不会买的股票（案例锚定）
3. 表达风格（语言特征、常用句式）
4. 内在矛盾（保持诚实，不脸谱化）

### 对各 change 的约束

| Change | 约束 |
|--------|------|
| L3 foundation | 单 agent prompt 按 Level 2 四层结构编写（决策框架/案例/风格/矛盾） |
| L3 full council | 全天团 prompt 按 Level 2 四层结构编写，不引入 RAG |

## AD-09 · L3 拆分以辩论增量为 gate

**状态**：已确认  
**来源**：total-design §8（1193-1195 行）

### 问题

L3 为什么拆成两个 change（foundation + full council），而不是合并成一个？

### 决策

**L3 先做单 agent 验证架构，再扩全天团。拆分依据是「辩论增量」gate。**

迭代原则：
1. 先把 L1+L2 做好（选股漏斗），这部分的 ROI 最高
2. L3 天团先从 1 个 agent（巴菲特）开始深研，验证 debate.py 编排、AgentOutput schema、校准流程
3. 边加 agent 边看辩论质量有没有信息增量
4. 如果 6 个 agent 说的话和 1 个没区别，就别加到 5 个

**gate 条件**：`l3-council-foundation` 完成后，必须同时满足机制门和质量门，才进入 `l3-full-analyst-council`：

**机制门（必要前提）**：
- debate.py 编排能正常运行完整流程（单 agent 场景下 Round 1 可独立运行，Round 2-4 的框架代码可执行）

**质量门（至少一项为真）**：
- 单 agent 深研报告比 L2 Scout 有显著信息增量
- 校准测试（§6.6）立场一致性过关

如果机制门不通过，说明编排实现有 bug；如果机制门通过但质量门不通过，说明 prompt 或校准有问题——两种情况都应暂停扩 agent，先修复再评估。

### 对各 change 的约束

| Change | 约束 |
|--------|------|
| L3 foundation | 只实现巴菲特单 agent + debate 骨架，验证架构可行性 |
| L3 full council | 必须在 foundation 通过 gate 后才开始；按辩论增量原则逐 agent 扩展 |
